Step 2 - Chunking Audit

Chunking strategies implemented:
- Fixed chunking.
- Recursive paragraph-aware chunking.
- Semantic sentence-group chunking.

Chunk size:
- Fixed: 400 words.
- Recursive: max_words=400.
- Semantic: 5 sentences per chunk.

Chunk overlap:
- Fixed: 50 words.
- Recursive: no explicit overlap; paragraph-based accumulation.
- Semantic: no explicit overlap; sentence grouping.

Recursive rules:
- Split input by paragraphs.
- If a paragraph exceeds max_words, split it into max_words slices.
- Accumulate paragraphs until the max_words limit is reached, then flush a chunk.

Semantic chunking method:
- Sentence boundaries are detected with a regex that respects punctuation.
- Sentences are grouped into fixed blocks of 5 sentences per chunk.

Which chunking used in final system:
- The current benchmark and Streamlit default use semantic chunking as the primary runtime strategy.
- The ablation study shows fixed chunking has the highest faithfulness in the latest run, but the deployed/default path remains semantic in the code path used for the evaluated system.

Why chosen:
- Semantic chunking is the coded default for the current benchmark and UI flow.
- It provides coherent sentence-group chunks and works well with the hybrid retrieval setup.

Step 5 - Retrieval Pipeline Audit

Full pipeline:
- Query -> query embedding -> hybrid retrieval -> rerank (optional) -> generation

Semantic retrieval model:
- BAAI/bge-small-en-v1.5

BM25 implementation:
- rank-bm25 BM25Okapi over tokenized local chunk text.

Hybrid formula:
- score = semantic_weight * semantic_norm + bm25_weight * bm25_norm
- Final adjusted score = score * source_multiplier(metadata)

Normalization method:
- Min-max normalization is applied independently to semantic scores and BM25 scores.

Weights used:
- Best sweep config: semantic_weight=0.3, bm25_weight=0.7
- Final submission evaluation uses semantic_weight=0.3, bm25_weight=0.7

top_k value:
- top_k=6 in the final 20-query evaluation.

Notes on retrieval behavior:
- The hybrid search gathers candidates from semantic and BM25 paths.
- It keeps source diversity by preferring one hit per source where possible.
- It filters noisy chunks and near duplicates.

Step 8 - Faithfulness Evaluation Audit (MANDATORY)

LLM judge model:
- Primary: mistralai/Mistral-7B-Instruct-v0.2 via Hugging Face Inference API.
- Local fallback: google/flan-t5-small.

Claim extraction method:
- Sentence splitting with regex over punctuation, newlines, and semicolons.
- Very short fragments are filtered out.
- If no claims remain, the full fallback text is used.

Prompt used:
- System: Return only one label: SUPPORTED or NOT_SUPPORTED.
- User prompt structure:
  - You are an expert evaluator.
  - Claim: <claim>
  - Context: <retrieved_context>
  - Is this claim supported by the context?
  - Answer only SUPPORTED or NOT_SUPPORTED.

Scoring method:
- Faithfulness score = supported_claims / total_claims.
- Refusal answers are scored as 0.0 with a refusal_guardrail detail.

Example claim verification:
- Example claim: The efficient frontier is a set of optimal portfolios that offer the highest expected return for a defined level of risk, or the lowest risk for a given level of expected return.
- Verdict: SUPPORTED
- Backend: chat.completions
